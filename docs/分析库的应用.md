这些依赖看起来和 RAG/向量检索相关。我先在项目里查一下它们具体用在哪。

这些依赖组合起来，典型用途是 RAG（检索增强生成）知识库问答。你项目里的 RoadMap.md 也写了「引入知识库 RAG，进行检索」，和这组库是对上的。

下面逐个说明:

langchain-community (>=0.3, <1.0)
LangChain 的社区集成包，提供大量第三方组件的封装，例如：

文档加载器（PDF、Word、网页等）
文本分割器
各类向量库、搜索引擎的适配器
做 RAG 时，常用来读文档、切块、接外部服务，是 langchain 核心包之外的扩展生态。

langchain-chroma (>=0.1, <1.0)
LangChain 对 Chroma 向量数据库的官方集成.

作用：把文本转成向量后存进 Chroma，并在用户提问时做相似度检索，找出最相关的文档片段，再交给 LLM 回答。适合本地、轻量的知识库场景。

langchain-huggingface (>=0.1, <1.0)
LangChain 对 Hugging Face 生态的集成，RAG 里最常见的是：

Embedding 模型（把文本变成向量）
本地或 HF 上的开源模型
若不想用 OpenAI/DeepSeek 的 embedding API，可以用这个包加载 sentence-transformers 等本地模型做向量化。

chromadb (>=0.5, <1.0)
Chroma 向量数据库本体（langchain-chroma 底层会用到它）。

功能：存储文档向量、建索引、按语义相似度查询。数据可持久化到磁盘，适合「上传文档 → 建库 → 问答」这类流程。

sentence-transformers (>=3.0, <4.0)
把句子/段落编码成稠密向量的库，基于 PyTorch。

RAG 里通常负责 embedding（例如 all-MiniLM-L6-v2），把用户问题和知识库文档映射到同一向量空间，才能做语义检索。常与 langchain-huggingface 一起用。

python-multipart (>=0.0.9, <1.0)
FastAPI 处理文件上传（multipart/form-data）时需要的依赖.

若要做「上传 PDF/TXT 建知识库」，接口里会有 UploadFile，没有这个包 FastAPI 无法正确解析上传请求.

整体关系（RAG 流程）
用户上传文档
    ↓
langchain-community（加载 + 切分文档）
    ↓
sentence-transformers / langchain-huggingface（文本 → 向量）
    ↓
chromadb + langchain-chroma（存储 + 检索）
    ↓
用户提问 → 检索相关片段 → LangChain + DeepSeek（生成回答）
上传接口侧还需要 python-multipart.
